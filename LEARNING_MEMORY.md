## Current mental model

Synopsys Virtualizer appears in this document set as a layered platform rather than a single tool:

- `Virtualizer Studio` is the umbrella workspace and automation surface.
- `TLM Creator` is the model-authoring layer for building blocks, interfaces, memories, registers, behaviors, callbacks, tasks, and packaging metadata.
- `VDK Creator` is the platform-assembly layer for composing complete virtual development kits from models and libraries.
- `VP Explorer` is the runtime, debug, tracing, and analysis environment centered around `VP Projects` and `VP Configs`.
- `Simulation Scripting` and `Simulation Probes` are the in-simulation Python automation layer for influencing behavior from inside the running simulation.
- `Embedded Software Debugger` integration connects the virtual platform to external software-debug tools.
- `Co-Simulation with Third-Party HDL Simulators` extends the flow across SystemC and HDL boundaries.
- `VP Profiling` provides the performance methodology and tooling.
- `MultiNode VDKs` appears to cover multi-node/distributed platform composition.
- `SystemC` is a foundational prerequisite, but the supplied manual is currently unreadable.

## Key findings extracted so far

### VP Explorer is the operational center

- `VP Project` files contain the platform name, description, and available builds.
- The manual says the `VP Project` file is generated by Platform Creator.
- `VP Config` files define how a platform is run: pre-run VP script, simulation executable, arguments, working directory, and related configuration tabs such as parameters, images, analysis, debug, and environment.
- VP Explorer supports both GUI and batch flows, including command-line launching and Tcl-based VP Config manipulation.

### Simulation Scripting is the inside-the-simulation automation model

From the manual's own distinction:

- `Simulation Scripting` works from the inside of the simulation process.
- `VDK Debug CLI` works from the outside for debugging and analysis.
- Simulation scripting is faster for in-process interaction.
- VDK Debug CLI can still execute while the simulation is suspended.

Primary scripting use cases called out in the guide:

- Embedded software configuration, monitoring, and control
- Model monitoring and control
- Test stimuli generation and fault injection
- Analysis and logging
- Modeling and prototyping
- Connections with third-party simulation tools

Important API surface areas mentioned in the guide:

- `sim` module for simulation control
- `CoreProbe` for core-centric probing
- `GDAProbe` and `FastTrackProbe` for analysis/tracing
- Startup scripts and standalone SystemC thread scripts
- Script libraries for reusable packaged features

### Profiling is treated as a first-class engineering discipline

The profiling guide frames performance work as essential for:

- Faster regression and parameter-sweep turnaround
- Better interactive software bring-up experience
- Creating room for more model detail without losing usability

The documented profiling progression is:

1. Benchmark correctly
2. Use light-weight profiler views
3. Use top-down profiling to isolate hotspots
4. Investigate SystemC process cost, kernel overhead, and dynamic quantum efficiency
5. Use statistical profilers such as `VTune`, `perf`, or `Visual Studio` when needed

Important metrics explicitly called out:

- Elapsed real time
- Real time factor
- Instruction throughput
- Quantum efficiency

### Co-simulation has strict structural rules

The co-simulation guide states:

- The top level of the co-sim design hierarchy must be `SystemC`.
- All HDL models in the design are simulated with one selected HDL simulator.
- The entry point is `SystemC Shell`, not the HDL simulator.
- Mixed-language nesting is allowed, but you must specify the maximum number of language switches.
- Port mapping, type mapping, and generic/parameter mapping are central concepts.
- The flow includes automatic HDL proxy module generation and a `SystemC Proxy Generator`.
- Co-sim debugging can still be performed in `VP Explorer`.

Environment variables explicitly mentioned for simulator setup:

- `VCS_INSTALLED_AT`
- `VCSI_INSTALLED_AT`
- `NC_INSTALLED_AT`
- `MTI_INSTALLED_AT`

### External embedded-software debugger support is broad

- `iSYSTEM winIDEA`

### Python automation is extensive

Notable signals from the Python references:

- `Virtualizer Studio Python Interface Reference Manual` spans Studio, VDK Creator, MultiNode, and TLM Creator APIs.
- `VP Explorer Python Interface Reference Manual` includes the `vpx` module plus DAI and database-writer interfaces.
- `Simulation Probes Python Interface Reference Manual` exposes rich runtime instrumentation across cores, signals, memories, sockets, observers, and utility helpers.
- The VP Explorer Python reference explicitly mentions support for `Python 3.9.1` and notes migration from older Python 2-based releases.

### BSP / ELF / IDE findings

Based on the searchable manuals, the documented flow is centered on building the virtual platform and then loading or debugging already-built software images inside the simulation.

What is explicitly documented:

- `Virtualizer Studio Python Interface Reference Manual` documents `create_vdk`, `create_fixed_vdk`, `set_active_compiler`, `get_supported_compilers`, platform `build()`, and `create_vp_config()`.
- The documented `build()` call is for the `platform simulation`, not for compiling embedded target software into an application ELF.
- `Virtualizer Studio - User Guide` explicitly describes Virtualizer Studio as an `Eclipse-based IDE` and a single environment for developing, debugging, and running both TLM models and VDKs.
- The same guide also states that standard `CDT Build Console` support is present, including `embedded software projects`, and that `GDB Hardware Debugging` is used to debug embedded software running inside the simulation.
- `VDK Creator - User Guide` states that VDK Creator designs, extends, and builds VDKs, and that the `build output of a VDK is a Virtual Prototype`.
- `VDK Creator - User Guide` also states that VDKs are designed, built, and run from within Virtualizer Studio, and that launching a VDK uses a `VP Config`.
- `VP Explorer` provides an `Images` tab and a `load_image` command for loading binaries, symbols, and A2L information into the virtual prototype before or during simulation.
- The `Images` tab documentation explicitly states that an `ELF` file can contain both the compiled software image and debug symbols.
- `VP Explorer` also documents OS-aware software analysis via `OS Kits`, including a generic Linux OS kit for Arm enabled through the `-ugLinux26ARMOSKit` linker flag in Platform Creator.
- `Simulation Scripting` documents startup-script use cases such as software image loading, device-tree generation, OS-kit configuration, and software re-initialization after reset.
- `VDK Creation Methodology - Manual` documents that Virtualizer Studio can handle standard `Eclipse CDT` projects for embedded C/C++ software tests and infrastructure libraries in the same workspace as TLM/VDK projects.
- The methodology guide further documents `GNU for Arm Eclipse` plugins as the recommended route for `create, build, debug, and manage Arm and AArch64 projects` within Virtualizer Studio.
- The methodology guide gives a concrete flow for rebuilding `PlatformTest.elf` using CDT tools, then configuring that local `.elf` as the VDK image through the `VP Config` `Images` tab, and then attaching `GDB`.
- `Embedded Software Debuggers` documents IDE/debugger integrations that consume software images from the VP Config or debugger setup:
  - Virtualizer Studio automated `GDB` launch uses the Eclipse Launch framework, Eclipse CDT, and software-image information from the VP Config.
  - MetaWare `launch_mdb_win32_load_image` can download software images to ARC cores in the target simulation.
  - `IAR Embedded Workbench` supports `Download and Debug` versus `Debug without Download` against the Virtualizer simulation.

What I did not find in the searchable manuals:

- No explicit `BSP` or `Board Support Package` workflow.
- No clear evidence that the VDK tool itself is a full embedded-software IDE that natively builds application `.elf` files for the target software.
- No documented flow for deploying to a physical hardware board; the documented download/debug flows are to the `Virtualizer simulation` or `platform`, not to a real board.

Working interpretation:

- `Virtualizer/VDK` is documented here primarily as a virtual-platform creation, build, run, debug, and analysis environment.
- The software side is mostly treated as images plus symbols that are loaded into or debugged on the simulated target, but there is documented IDE support for certain embedded-software projects inside the same Eclipse workspace, especially `PlatformTest` and Arm/AArch64 integration-test flows.
- The strongest documented `.elf` build flow is for `integration testing` and `PlatformTest`, not for a generic BSP-managed product software build flow.
- For non-Arm VDKs, the methodology guide says `PlatformTest` must be ported to the respective VDK and processor toolchain.
- If a BSP/package or physical-board handoff flow exists, it is likely in the non-searchable manuals or outside this repo's currently readable documentation.
